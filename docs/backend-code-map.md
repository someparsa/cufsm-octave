# Backend Code Map

This document maps the backend Octave/MATLAB files in the repository. It is intended for developers who need to understand where the numerical routines, JSON runner, helpers, plotting code, and retained upstream modules live.

The repository currently retains a broad portion of CUFSM source. Not every retained file is used by the JSON runner today. The JSON workflow uses a narrower path through the backend, described first.

## Primary JSON Execution Path

The preferred workflow is:

```bash
octave-cli --quiet cufsm_json.m examples/lipped-channel.json
```

Main call chain:

| Step | File | Role |
| --- | --- | --- |
| 1 | `cufsm_json.m` | Public script wrapper. Reads command-line arguments, adds `helpers/`, and calls `cufsm_json_run`. |
| 2 | `helpers/cufsm_json_run.m` | JSON runner. Parses input, validates model tables, applies mesh refinement and loading, builds analysis settings, calls `stripmain`, classifies modes, and writes JSON/text results. |
| 3 | `analysis/stripmain.m` | Main finite strip eigenvalue solver for signature-curve analysis. Assembles stiffness/geometric stiffness matrices and returns `curve` and `shapes`. |
| 4 | `analysis/cFSM/*` | cFSM basis, constraints, and mode-classification utilities used for modal participation. |
| 5 | `analysis/cutwp_prop2.m`, `analysis/yieldMP.m`, `analysis/stresgen.m` | Section properties, yield reference actions, and stress generation for `generated_from_actions` loading. |
| 6 | `helpers/cufsm_json_run.m` | Post-processing: signature curve, local minima, family minima, member-length participation, signature-minima participation, JSON/text reports. |

| File | Current role |

## Python Package Files

The `cufsm_octave/` package is a thin Python layer around the JSON workflow. It
does not reimplement the Octave solver. It prepares JSON inputs, launches the
Octave runner, reads result JSON, and provides optional post-processing helpers.

| File | Role |
| --- | --- |
| `cufsm_octave/__init__.py` | Public Python package exports. |
| `cufsm_octave/io.py` | JSON loading/writing helpers and Python-side input validation. Uses `jsonschema` when the optional dependency is installed, with lightweight fallback checks otherwise. |
| `cufsm_octave/runner.py` | Subprocess wrapper around `octave-cli --quiet cufsm_json.m input.json`; returns `CufsmResult` and raises `CufsmRunError` on non-zero Octave exit. |
| `cufsm_octave/result.py` | `CufsmResult` wrapper for result JSON, with accessors for signature curves, critical points, family minima, mode participation, and optional pandas DataFrame conversion. |
| `cufsm_octave/templates.py` | JSON input builders for common section templates, custom centerline models, materials, loading blocks, length definitions, cFSM defaults, and signature-curve analysis blocks. |
| `cufsm_octave/plotting.py` | Optional matplotlib post-processing helpers for plotting and saving signature curves from result JSON. |

| Helper | Role |
| --- | --- |

Supported section-template labels:

| Label | Geometry represented |
| --- | --- |
| `unlipped-channel` | Web plus two plain flanges. |
| `lipped-channel` | Web, two flanges, and lips at the flange tips. |

| `sigma-section` | Lipped channel with the web-stiffener path documented in `docs/python-tooling.md`. |
| `stiffened-web-channel` | Channel with the central web-stiffener path documented in `docs/python-tooling.md`. |

## Core Analysis Files

These files implement the main finite strip stiffness, geometric stiffness, assembly, boundary-condition, spring, and solver workflow.

| File | Role |
| --- | --- |

| `analysis/assemble.m` | Assembles element stiffness and geometric stiffness into global matrices. |
| `analysis/elemprop.m` | Computes strip element width and orientation from node and element tables. |
| `analysis/klocal.m` | Local elastic stiffness matrix for strip elements. |
| `analysis/kglocal.m` | Local geometric stiffness matrix for strip elements. |

## Files That Should Be Treated Carefully

- `.asv` files are MATLAB autosave files and should not be treated as active source.
- `.mat` files are binary data resources from upstream workflows.
- `.fig` files are MATLAB figure resources.
- Plotters and GUI files are retained but are not part of the headless JSON runner.
- Some routines exist in both `analysis/` and `helpers/` because of upstream history. The JSON path currently adds both directories to the Octave path; avoid introducing new duplicate routines.

## Practical Developer Notes

- For new headless features, prefer extending `helpers/cufsm_json_run.m` and documenting the JSON contract.
- For numerical changes, identify whether the active path uses the `analysis/` version or a helper copy before editing.
- Keep GUI/waitbar code out of the JSON path. If an upstream routine opens UI, create a headless wrapper or refactor carefully.
- Keep JSON output tabular where possible. Numeric tables are easier for Octave, Python, spreadsheets, and validation scripts to consume.
- Preserve upstream attribution in comments and docs when moving or adapting numerical routines.
